Examples

Elliptical web components is a set of library packages that implement the W3C web component standard. It plays well other libraries, including other component implementations such as Polymer. Indeed this page uses a Polymer component(mark-down) for code syntax highlighting. This modular structure allows elliptical web components to be implemented separately as behaviors or mixins with libraries such as Polymer.

    Documentation

    Element Factories

    The Elliptical-component library package defines two factories, $.element and $.component, that implement the web component standard over an extension of the jquery ui widget factory. The difference between the two is that $.component extends the $.element base factory with observable, two-way data-binding.

    ```js $.element([Element],tag,[mixins],prototype); $.component([Element],tag,[mixins],prototype); ```
    NOTE: The prototype object passed to the elliptical component factories above lives in the jquery object prototype chain. It will be referred to as the Component Prototype. This differs from the Custom Element Prototype which lives on the DOM Object prototype chain. Elliptical exposes public properties and methods on the component prototype to the custom element prototype. The converse, however, is not true. That is, if public properties and methods are defined directly on the custom element prototype, they will not be available on the component prototype.

    Parameters

    Element {Document Object} [optional] HTML Element Type default: HTMLElement
    tag {String} required tag name default:null
    mixins {Array} [optional] object array of mixins default:null
    prototype {Object} required methods and props for the element definition default:undefined

    Lifecycle Callbacks

    Elliptical web components exposes the 4 W3C defined lifecycle callbacks in the following manner.

    ```js $.element('my-tag',{ //created,attached _initElement:function(){ }, //attributeChanged _attributeChanged:function(attrName,oldValue,newValue){ }, //detached _detached:function(){ } }); $.component('my-tag',{ //created,attached _initComponent:function(){ }, //attributeChanged _attributeChanged:function(attrName,oldValue,newValue){ }, //detached _detached:function(){ } }); ```

    AttributeChanged and Detached are also available directly on the custom element prototype chain.

    ```js var myElement=document.querySelector('my-element'); myElement.attributeChanged=function(attrName,oldValue,newValue){ }; myElement.detached=function(){ }; ```

    Properties and Attributes

    Public properties can be declared via element attributes or imperatively defined in the options property on the component prototype(a la jquery plugins). Getters and setters are defined for public properties on both the component prototype chain and the custom element(DOM) prototype chain. ```html <my-element foo="bar"></my-element> ``` ```js $.element('my-element',{ options:{ myProp:'test' }, _initElement:function(){ var foo=this.foo; //-->'bar' var myProp=this.myProp; //-->'test' } }); var myElement=document.querySelector('my-element'); var foo=myElement.foo; //-->'bar' ```

    Reserved Public Properties

    scope {String} Specifies a property in the current ViewData context to assign to the component $scope. $scope[scope]=$viewBag[scope] OR: if a local data context is set, defines the property map to the component $scope $scope[scope]=data $.component
    data {Object|Array} the stringified local data context assigned to the component $scope. If data is an object, $scope is set equal to data, unless a scope attribute is set. In that case, $scope[scope]=data. If data is an array, the scope attribute must be set, otherwise data is ignored. $.component
    $scope {Object} The Observable, template-bound property $scope MUST be an OBJECT $scope can be deep(nested) with arbitrary property types(.e.g, arrays) default:{} $.component
    $viewBag {Object|Array} returns the current ViewData context $.component
    objectAssign {Boolean} assigns the 'scope' property on the ViewData context directly to the $scope $scope=$viewBag[scope] default:false $.component
    scopeBind {Boolean} scopeBind set to false disables $scope as an observable default:false $.component
    autoRebind {Boolean} Auto-rebinds template on array inserts/deletes default:true $.component
    service {String} Service name for service location/dependency injection default:null $.component
    template {undefined} attribute that defines the template DOM for a component (i)the inner html is compiled and rendered as a dustjs template (ii) the template DOM is bound to the observable $scope (iii)the template attribute can be set in the definition or instance (iv) the attribute has no value assigned $.component

    Methods

    Methods prefaced with an underscore "_" are considered private methods and are only available on the component prototype. Public methods are available on both component and custom element prototype chains.

    ```js $.element('my-element',{ log:function(msg){ console.log(msg); } }); var myElement=document.querySelector('my-element'); myElement.log('test'); //--> 'test' ```

    Events

    Add event listeners to the component by providing a events object that maps events to event handler function names.

    You can also add an event through the private method, this._event

    ```js $.element('my-element',{ _events:{ 'input@focus': '_onFocus' }, _onFocus:function(event){ } }); ``` ```js $.element('my-element',{ _initElement:function(){ this._event(this.element,this._data.click,'input',this._onFocus.bind(this)); }, _onFocus:function(event){ } }); //syntax _events:{ 'selector@event':'handler' } //if selector is omitted, the event is dispatched on the custom element //selector is a descendant selector, the parent is the custom element //UNLESS: //selector=window or document ```

    this._event(parent,event,[selector],handler)

    parent {Object} The DOM or parent DOM the listener is attached to
    event {String} event name NOTE: this._data.click=desktop/mobile click this._data.press=desktop/mobile press
    selector {String} descendant selector Optional
    handler {Function} The event handler
    Annotated Event Handling

    To add a click listener to local DOM children, use the 'on-click' annotation in your template.

    ```html <span on-click="_onClick">Click</span> ```

    PubSub

    $.component exposes an additional pubsub system apart from the DOM for event listeners.

    ```js $.component('my-element',{ _subscriptions:{ 'my.channel':'_channelHandler' }, _channelHandler:function(data){ } }); ```

    You can also subscribe to a channel through the private method, this._subscribe.

    You can publish to a channel through the private method, this._publish.

    ```js this._subscribe('my.channel',this._handler.bind(this)); _handler:function(data){ } this._publish('my.channel',{test:'test'}); ```

    this._subscribe(channel,handler)

    channel {String} channel name $.component
    handler {Function} channel handler $.component

    this._publish(channel,data)

    channel {String} channel name $.component
    data {Object} channel data $.component

    Private Data Store

    The component prototype exposes a private data store, this._data. ```js this._data.set('my-key',{test:'test''); var myData=this._data.get('my-key'); //--->{test:'test'} ```

    this._data

    get param:key {String} returns: {*} $.element $.component
    set param:key {String} param:value {*} returns: {void} $.element $.component

    Data-Binding

    Elliptical data-binding maps a deep observable property($scope) on the element prototype chain to the component template DOM marked by a template attribute. Elliptical components use dustjs for template markup. Two-way data-binding is marked up using the data-bind attribute.

    The philosophy behind elliptical's approach is that deep observable data-binding should just work without imperative wiring. Otherwise your component is tied too closely to the model and you do not have--by definition--"reusable components."

    NOTE: The template attribute can be set on the custom element itself or on a child element.

    FURTHER NOTE: The template attribute can be set in the definition or the instance
    ```html <input type="text" data-bind="value:firstName" value="{firstName}"> <!--arrays--> {#model} <input type="text" data-bind="value:model.$0.firstName" value="{firstName}"> {/model} <!--deep nesting--> {#accounts} <span data-bind="text:accounts.$0.name">{name}</span> <flex-box class="column"> {#phoneNumbers} <input type="text" data-bind="value:phoneNumbers.$1.phone" value="{phone}"> {/phoneNumbers} </flex-box> {/accounts} ```
    Data-Bind Syntax
    ```js data-bind="type:path" ```

    "type" can either be "text" for textNode(span), or an attribute type, such as "value."

    path is dot-notation from parent to property. For nested array properties and sub-properties, it is by index.

    Path Index Helpers

    Nested objects with array properties and sub-properties requires path indexes to properly two-way data-bind to the template DOM. Elliptical extends the dustjs library to offer path index helpers. $<n>, where n=0,1,...j indicates the depth.

    Path Index Helpers

    $0 depth: 0 model.$0.prop ex: accounts.$0.firstName $.component
    $1 depth: 1 model.$0.prop.$1.subprop ex: accounts.$0.addresses.$1.street $.component
    $2 depth: 2 model.$0.prop.$1.subprop.$2.subsubprop ex: accounts.$0.addresses.$1.phoneNumbers.$2.phone $.component
    . $.component . .
    Annotated attribute binding

    If the custom element definition is marked with a parse-attr attribute, then you can use [[ ]] to bind attribute values in the instance to placeholders in the definition.

    ```html <!--definition--> <my-element parse-attr definition> <template> <span title="[[title]]">[[text]]</span> <content></content> </template> </my-element> <!--instance--> <my-element title="My Title" text="Welcome"></my-element> ```

    Data Sources

    Data-bound components need data. The question becomes how do we get the data. You can manually wire-up the data in the created callback init event, but in the real-world this violates re-usability. In the MVC design pattern, web components are part of "the View," and the View generally shouldn't be fetching the model. This violates a single-responsibility principle. It is the job of the Controller to pass the model to the View. Given this, if the MVC framework, server-side or client-side, binds the model to a ViewData context(in elliptical, underneath the covers, this would be window.__viewData), then we can use declarative markup in the attributes to bind a data-model to our web components. This is the purpose of the "scope" attribute. If a web component has the scope attribute set, it will try to fetch its model from that property in the ViewData context(which it will then bind to the $scope observable). NOTE: it is easy in such frameworks as ASP.NET(using a HtmlHelper) or ExpressJS to serialize a model to the global context in a script tag. In client-side frameworks, it is trivial to wire a render method to bind the model to the global context.

    Still, there will be many times where you might want to encapsulate the service request inside the component. In such cases, we can rely on a dependency injection/service locator pattern.

    Data Sources

    manual hard-coded in a component method, typically init imperative $.component
    [data] attribute json string passed to the component's 'data' attribute annotation/declarative $.component
    ViewData context bind to a property on the current ViewData context window.__viewData annotation/declarative $.component
    Dependency Injection inject the callable service into the component a mix of declarative & imperative $.component

    Dependency Injection

    By default, elliptical components uses elliptical-container as the DI container. The container is passed to the component prototype chain to implement a service locator. We use public attributes on the custom element(.e.g, 'service') to locate the service.

    ```html <my-element service="MyService"></my-element> ``` ```js $.component('my-element',{ _initComponent:function(){ this._bindService(); }, _bindService:function(){ var self=this; var serviceName=this.service; var myService=this.$service(serviceName); myService.get({},function(err,data){ self.$scope=data; }); } }); ```
    You should not dash format service attribute values.

    Service Locator

    $service param:{String} name returns a service(singleton,constructor,prototype,class) returns: {*} $.element $.component
    $serviceAsync param:{String} name param:{Function} callback async returns a service(singleton,constructor,prototype,class) $.element $.component
